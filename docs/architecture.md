# Office AI 오케스트레이션 설계

## 목표

사용자는 하나의 명령 채널만 사용합니다. 시스템은 명령을 업무 유형으로 분류하고, 작업을 원자 단위로 분해하고, 각 작업에 필요한 최소 성능의 모델을 선택하며, 독립 검증을 통과한 결과만 보고합니다.

핵심 원칙은 “처음부터 가장 비싼 모델을 쓰지 않는다”가 아니라 “실패 비용까지 포함한 기대 총비용이 가장 낮은 경로를 선택한다”입니다. 저가 모델의 반복 실패가 프리미엄 모델 1회보다 비싸면 즉시 상향합니다.

## 실행 흐름

```mermaid
flowchart LR
    A[단일 명령 채널] --> B[업무 분류기]
    B --> C[계획 생성기]
    C --> D[작업 DAG]
    D --> E[모델 라우터]
    E --> F[역할별 실행기]
    F --> G[결정론적 검사]
    G --> H[독립 검증기]
    H -->|통과| I[보고서 생성]
    H -->|수정 가능| F
    H -->|고위험/반복 실패| J[상위 모델 또는 사람 승인]
    J --> I
```

## 역할

| 역할 | 책임 | 기본 산출물 |
|---|---|---|
| 접수/분류 | 업무 유형, 위험도, 도구 권한 판정 | `TaskEnvelope` |
| Planner | 목표, 제약, 완료 조건, 작업 DAG 작성 | `Plan` |
| 기획 | 리서치, 요구사항, 우선순위, 정책 작성 | 근거가 연결된 기획서 |
| 개발 | 코드 변경, 테스트, 빌드 결과 작성 | 패치와 실행 로그 |
| PM | 의존성, 일정, 승인 지점, 진행 상태 관리 | 상태 보고 |
| Verifier | 원문·테스트·정책과 산출물 교차검증 | `VerificationReport` |
| Reporter | 결정 사항, 결과, 미해결 위험만 요약 | 최종 보고 |

## 모델 라우팅

### 입력 특성

- 업무 종류: 분류/요약/리서치/기획/코드/검증
- 복잡도: 입력 길이, 의존성 수, 예상 단계 수
- 위험도: 외부 전송, 비용 발생, 배포, 법무/보안 영향
- 검증 가능성: 테스트나 스키마로 자동 판정 가능한 정도
- 실패 이력: 동일 작업의 재시도 횟수와 실패 원인

### 계층

| 계층 | 적합한 작업 | 제한 |
|---|---|---|
| 경제형 | 분류, 추출, 포맷 변환, 짧은 요약 | 외부 근거 판단과 최종 승인 금지 |
| 표준형 | 기획 초안, 일반 코딩, 문서 작성, 도구 호출 | 고위험 최종 판단 금지 |
| 프리미엄 | 복잡한 설계, 모호한 문제, 최종 반박 검증 | 처음부터 기본값으로 사용하지 않음 |

### 상향 조건

1. 스키마 검증 2회 실패
2. 근거 충돌 또는 인용 불일치
3. 계획의 의존성 수가 임계값 초과
4. 보안·금전·배포 등 고위험 동작
5. 검증기가 낮은 신뢰도 또는 반례를 반환

단순히 모델이 “확신한다”고 말하는 값은 라우팅 신호로 쓰지 않습니다.

## 토큰 절감 구조

1. 전체 대화 대신 `TaskEnvelope`만 다음 단계에 전달합니다.
2. 원문은 저장소에 보관하고 에이전트에는 필요한 청크와 인용 ID만 제공합니다.
3. 계획, 산출물, 검증 결과를 각각 구조화 JSON으로 유지합니다.
4. 동일 자료의 요약·임베딩·도구 결과는 콘텐츠 해시로 캐시합니다.
5. 작업 DAG의 독립 노드는 병렬 실행하고, 합치는 단계에 결과 요약만 전달합니다.
6. 보고 단계는 로그 전체가 아니라 결정, 증거, 실패, 비용 집계만 읽습니다.
7. 프롬프트는 역할별 고정 규칙과 업무별 가변 입력을 분리해 프롬프트 캐시 적중률을 높입니다.

## 검증과 할루시네이션 방지

검증은 생성 모델의 자기 평가 하나로 끝내지 않습니다.

- 구조 검증: JSON Schema, 타입, 필수 필드, 허용 값
- 코드 검증: lint, typecheck, unit/integration/E2E test, diff scope
- 사실 검증: 각 주장에 `source_id`, 원문 위치, 확인 시각 저장
- 상호 검증: 실행기와 다른 모델/프롬프트가 반례를 찾음
- 정책 검증: 외부 전송, 배포, 결제, 삭제는 명시적 승인 필요
- 종료 조건: 완료 기준마다 기계 판정 가능한 evidence가 있어야 함

`VerificationReport`에는 `passed`, `failed_checks`, `evidence`, `residual_risk`, `recommended_action`을 저장합니다. 근거가 없는 문장은 최종 보고에서 제거하거나 “추정”으로 명시합니다.

## 권장 데이터 계약

```ts
type TaskEnvelope = {
  taskId: string;
  category: "planning" | "development" | "pm" | "research" | "mixed";
  objective: string;
  constraints: string[];
  acceptanceCriteria: string[];
  risk: "low" | "medium" | "high";
  tokenBudget: number;
  costBudget: number;
  approvalPolicy: string[];
};

type WorkUnit = {
  id: string;
  role: string;
  dependsOn: string[];
  inputRefs: string[];
  expectedOutputSchema: string;
  modelTier: "economy" | "standard" | "premium";
  retryLimit: number;
};
```

## 구현 순서

1. 단일 명령함 + 업무 분류 + 작업 DAG
2. 모델 라우터 + 토큰/비용 원장
3. 기획·개발·PM 템플릿과 완료 기준
4. 코드/문서용 결정론적 검증기
5. 근거 저장소와 인용 강제
6. 승인 게이트와 외부 도구 연결
7. 실패 데이터 기반 라우팅 정책 튜닝

첫 버전은 자동 분류와 검증이 쉬운 개발 업무부터 연결하는 편이 좋습니다. 배포·메일 발송·문서 공유처럼 외부 상태를 바꾸는 작업은 승인 게이트가 준비된 뒤 활성화합니다.
