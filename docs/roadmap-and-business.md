# PART A. 완성도 진단

## A-1. 레이어별 완성도

| 레이어 | 완성도 | 근거 |
|--------|:---:|------|
| 코어 오케스트레이션 | **90%** | intake→dispatcher→orchestrator(DAG 동시성·크리틱 재작업 루프)→budget→ledger 전 구간 구현·테스트됨 |
| Provider 연동 | **85%** | openai/anthropic/lmstudio/cursor-agent-cli/codex-cli/mock + 재시도·티어 폴백. 실 연결 라이브 테스트는 없음 |
| 컨텍스트/기억 | **70%** | 토큰 예산 기반 컨텍스트 **선택**(context-pack)·work-profile·스캐너 있음. **압축/요약·장기기억은 없음** |
| Electron 통합 | **80%** | IPC·safeStorage·트레이/알림·스케줄러·진단/산출물 내보내기·라이선스·크래시(opt-in) |
| Renderer UI | **78%** | 8개 뷰 실데이터 연결 완료. 3D 오피스. 접근성/반응형/다국어 미흡 |
| 패키징/배포 | **40%** | electron-builder 설정·아이콘 존재. **코드사이닝·자동업데이트(G3)·notarization 미완** |
| 비즈니스/백엔드 | **20%** | 오프라인 서명 라이선스(G13)·직원 마켓 기획(G16)만. 서버/결제/발급 자동화 없음 |

**한 줄 요약:** *엔진은 제품급, 셸/UX는 양호, 배포·수익화 인프라는 초기 단계.*

## A-2. 강점 (지킬 것)

- **0토큰 규칙 기반 분류·분배**(intake·dispatcher) → 실제 토큰 절약이 측정되는 셀링포인트(savings 리포트).
- **이벤트 스트림 단일화**(event-bus→ledger→IPC→zustand): UI가 엔진 내부를 몰라도 됨. 데모/실엔진 자동 전환.
- **오프라인 우선**: sqlite 폴백, 로컬 모델(LM Studio), 서버 없이 서명 라이선스 검증.
- **테스트된 순수 로직**: 상태머신·라우팅·예산·redaction 등 회귀 안전망.

## A-3. 핵심 갭 / 리스크

1. **실제 "실행"이 없다.** 산출물은 텍스트(마크다운)까지. 파일 쓰기·git·브라우저·메일 등 **side-effect 실행 액션이 없음** → "업무를 대신 해준다"는 약속과 간극. (approval-gate에 side-effect kind는 있으나 실행기 미연결)
2. **컨텍스트 압축 부재.** 긴 대화/대용량 산출물 누적 시 토큰 폭증. 선택만 있고 요약·계층 압축·벡터 검색 없음.
3. **확장성(도구) 없음.** MCP/Skills/Hooks 등 사용자 확장 지점이 없어 기능이 고정됨.
4. **멀티 프로젝트/워크스페이스 개념 없음.** 모든 run이 평면적. 프로젝트별 맥락·예산·이력 분리 불가.
5. **배포 신뢰성.** 코드사이닝 없으면 SmartScreen 경고 → 구매 전환 저해. 자동업데이트 없음.
6. **수익화 인프라.** 키 발급·결제·검증이 수동. 규모화 불가.
7. **관측성.** 사용자 환경 오류 파악 수단이 진단 파일 수동 내보내기뿐(크래시는 opt-in).

## A-4. 즉시 개선 (Quick Wins, 1~3일 규모)

| 항목 | 이유 | 난이도 |
|------|------|:---:|
| 연결 **라이브 테스트** 버튼(API 핑 / CLI --version / LM Studio /models) | 지금 상태표시는 정적. "진짜 연결" 확인 UX | 하 |
| 3D 씬 번들 **코드 스플릿**(현재 OfficeScene 2.2MB) | 초기 로딩·메모리 | 하 |
| 에이전트/프로세스 뷰의 **빈 상태·로딩 상태** 정리 | 완성도 인상 | 하 |
| 오류 토스트/전역 에러 표면화(현재 콘솔) | 신뢰성 체감 | 중 |
| README/랜딩에 데모 GIF | 마케팅 | 하 |
| i18n 문자열 추출(ko 기본) | 향후 영어권 판매 | 중 |

# PART B. 기능 로드맵

각 항목: **목적 / 필요사항 / 중요지점 / 코어영향 / 난이도·우선순위**. 우선순위는 P0(먼저)~P3.

## B-1. 실제 실행 액션 (Action Runner) — **P0**
- **목적:** 산출물 생성에서 그치지 않고 승인된 작업을 실제 수행(파일 저장, git 커밋/PR, 셸 명령, HTTP 호출, 메일/메신저 발송).
- **필요사항:** 화이트리스트 기반 액션 레지스트리, 샌드박스/드라이런, 승인 게이트 연동(이미 side-effect kind 존재), 실행 로그·롤백.
- **중요지점:** 보안 최우선 — 액션은 **명시 허용된 것만**, 위험 액션은 강제 승인. 코어 dispatcher 재작성 금지 → **오케스트레이터 밖 실행 레이어**(electron 측)로 구현하고 결과를 이벤트로 주입.
- **코어영향:** 낮음(계약 추가 없이 electron 액션 핸들러 + IPC).
- **난이도:** 상 / **P0**.

## B-2. 컨텍스트 압축 & 장기기억 — **P0**
- **목적:** 긴 run·다수 산출물에서도 토큰 폭증 없이 맥락 유지.
- **필요사항:**
  - 계층 요약(맵-리듀스): 유닛 산출물 → run 요약 → 프로젝트 요약.
  - 슬라이딩 윈도우 + "중요 사실" 추출(기존 context-pack·token-estimator 확장).
  - 벡터 인덱스(로컬, 예: sqlite-vss/`better-sqlite3` + 임베딩) 선택적 도입.
  - 임베딩 provider 추가(로컬 임베딩 우선, 비용 0).
- **중요지점:** 압축은 **손실**이므로 원문 참조(citation) 유지. 요약도 토큰을 쓰므로 economy 티어로. 코어 orchestrator 테스트 깨지지 않게 **선택적 옵션**으로.
- **코어영향:** 중(context 모듈 확장은 추가형).
- **난이도:** 상 / **P0**.

## B-3. MCP 관리 (도구 확장) — **P1**
- **목적:** 사용자가 MCP 서버를 등록해 에이전트가 외부 도구(DB, 브라우저, 사내 API)를 사용.
- **필요사항:** MCP 클라이언트 내장(stdio/HTTP), 서버 목록 UI(추가/삭제/인증), 도구 카탈로그 노출, 승인 게이트 연동, 프롬프트에 도구 스펙 주입.
- **중요지점:** MCP 도구 호출 = side-effect → B-1 액션 러너와 통합. 자격증명은 safeStorage. 도구 오남용 방지(권한·레이트리밋). 코어는 provider가 tool-call을 반환하도록 하는 부분이 필요 → **새 계약 추가**이므로 별 트랙(코어 테스트 동반 확장) 필요.
- **코어영향:** 상(툴콜 루프는 orchestrator 확장). 신중히.
- **난이도:** 상 / **P1**.

## B-4. Skills 관리 (재사용 워크플로) — **P1**
- **목적:** 자주 쓰는 지시/절차를 "스킬"로 저장·공유·구매(직원 마켓과 시너지).
- **필요사항:** 스킬 스키마(트리거·프롬프트팩·필요 직원/도구·기본 승인정책), 스킬 CRUD UI, 실행 시 주입, 마켓 배포(서명).
- **중요지점:** 스킬 = 프롬프트+구성 오버레이. 코어 미변경(엔진에 config로 주입). 버전/호환성 관리. 악성 스킬 검증(서명·샌드박스).
- **코어영향:** 낮음(구성 주입형).
- **난이도:** 중 / **P1**.

## B-5. Hooks (이벤트 자동화) — **P2**
- **목적:** 특정 이벤트(run 완료/실패/승인요청/예산초과)에 사용자 정의 동작(웹훅·스크립트·알림·다음 run 트리거).
- **필요사항:** 훅 등록 UI, 이벤트 매처, 실행 샌드박스(B-1 재사용), 재시도/실패 로깅.
- **중요지점:** event-bus는 이미 모든 상태를 발행 → **electron 측 구독자로 훅 실행**하면 코어 무변경. 무한 루프 방지(훅→run→훅) 가드.
- **코어영향:** 없음.
- **난이도:** 중 / **P2**.

## B-6. 프로젝트/워크스페이스 관리 — **P1**
- **목적:** run·예산·기억·스케줄을 **프로젝트 단위**로 분리.
- **필요사항:** Project 엔터티(id·이름·루트경로·연결된 기억폴더·예산·직원 로스터), 프로젝트 선택기(상단), ledger 스키마에 projectId 컬럼, 뷰 필터.
- **중요지점:** ledger는 sqlite → 마이그레이션 필요(기존 데이터 projectId=default). intake의 프로젝트 추론(이미 work-profile에 유사 로직) 재사용. 코어 계약에 projectId를 **선택 필드**로 추가하면 회귀 최소.
- **코어영향:** 중(telemetry/ledger 스키마·계약 소폭).
- **난이도:** 중상 / **P1**.

## B-7. 자동 업데이트 (G3) — **P1**
- **목적:** 배포 후 원격 업데이트.
- **필요사항:** electron-updater + 배포 채널(GitHub Releases/S3), `electron-builder.yml`의 publish, "재시작하여 업데이트" 배너.
- **중요지점:** **코드사이닝 선행 필수**(미서명 시 Windows 경고·mac 차단). 롤백 전략.
- **코어영향:** 없음.
- **난이도:** 중 / **P1**.

## B-8. 코드사이닝 + 배포 신뢰성 (G14) — **P0(판매 전 필수)**
- **필요사항:** Windows EV 인증서, Apple Developer ID + notarization(사용자 구매). `docs/release-guide.md` 절차 존재.
- **중요지점:** 인증서는 **구매·갱신 비용** 발생(보고 대상). CI에서 서명 자동화.
- **난이도:** 중(대부분 행정) / **P0**.

## B-9. 관측성 / 분석 — **P2**
- **목적:** 사용량·오류·전환 파악.
- **필요사항:** opt-in 텔레메트리(익명), 크래시(Sentry 이미 배선), 제품 분석 이벤트(개인정보 배제), 서버 집계.
- **중요지점:** 개인정보보호법/GDPR — 기본 off·명시 동의·redaction 재사용.
- **난이도:** 중 / **P2**.

## B-10. 기타 백로그
- 협업/팀(공유 로스터·역할·감사로그) — P3
- 접근성(키보드/스크린리더)·다국어(i18n) — P2
- 웹/모바일 뷰어(읽기전용 보고서) — P3
- 프롬프트/모델 A/B·품질 회귀 추적 — P3
- 온디바이스 임베딩·완전 오프라인 모드 강화 — P2

# PART C. 비즈니스 모델

## C-1. 타깃 세그먼트
1. **1인 개발자/프리랜서·부트스트래퍼** (현재 UX 적합): BYOK로 비용 통제 + 토큰 절약이 핵심 가치.
2. **소규모 팀/에이전시**: 반복 업무 자동화(기획·리서치·보고), 직원 마켓·스킬.
3. **비개발 지식노동자**(장기): 로컬 기억 연동 + 쉬운 명령.

## C-2. 패키징 / 가격 (제안)

| 플랜 | 형태 | 포함 | 참고 |
|------|------|------|------|
| Free/체험 | 무료 | mock 무제한 + 유료 API 티어 10회(이미 구현) | 전환 유입 |
| Personal | **1회 구매**(영구) or 연구독 | 본체 + 기본 직원 3종 | 오프라인 서명 라이선스 재사용 |
| Pro | 구독 | + 프리미엄 직원·스킬·MCP·자동실행 | 지속 수익 |
| 직원/스킬 마켓 | 개별 구매 | 직무별 AI·워크플로 | G16 기획 완료 |
| Team | 시트 구독 | 공유·감사·SSO | 백엔드 필요 |

- **BYOK(키 직접) 기본** → LLM 원가 리스크 회피. 마진은 소프트웨어 라이선스에서.
- 선택: **프록시(게이트웨이) 요금제** — 사용량 마크업 수익(원가+ margin). 단, 크레딧/정산/남용 방지 백엔드 필요(C-4, D 참조).

## C-3. 현실적 수익 구조
- 초기(0~6개월): **Personal 1회 구매 + 직원 마켓**(오프라인 발급) → 서버 최소, 마진 높음.
- 성장기: **Pro 구독**(자동실행·MCP·스킬 가치) + 마켓 수수료.
- 확장기: **Team/게이트웨이** — 백엔드·결제·정산 본격화.

## C-4. 차별화 포인트(마케팅)
- "전부 premium이었다면 $X → 스마트 배정 Y, Z% 절약" (측정 가능한 절감).
- 로컬 기억 인용(내 업무 맥락 자동 포함).
- 3D 오피스(데모 GIF·바이럴 소재).
- 오프라인·BYOK(데이터 주권 — 규제/보안 민감 고객).

## C-5. 리스크
- 모델 가격 급변 → BYOK로 흡수.
- 플랫폼(Cursor/OpenAI) 정책 변화 → 멀티 provider로 분산(이미 확보).
- 미서명 배포 신뢰 문제 → B-8 선행.

# PART D. 백엔드 아키텍처

## D-1. 원칙: "필요할 때만 서버"
현재 **완전 로컬**이 강점. 서버는 *오프라인으로 못 하는 것*만 담당한다.

## D-2. 최소 백엔드가 필요한 기능
| 기능 | 서버 역할 | 대안(오프라인) |
|------|-----------|----------------|
| 라이선스/직원 키 발급 | 결제 확인→Ed25519 서명 키 발급 | 수동 발급(현재) |
| 결제 | PG 웹훅 수신·검증 | 외부 결제+수동 |
| 자동 업데이트 | 릴리스 아티팩트 배포 | 수동 다운로드 |
| 크래시/텔레메트리 | 수집·집계(opt-in) | 진단 파일 수동 |
| 게이트웨이(선택) | LLM 프록시·사용량 정산 | BYOK 직접 호출 |
| 팀 동기화(선택) | 공유 상태·감사 | 로컬 전용 |

## D-3. 스택 제안 (경량 우선)
- **서버리스 + 매니지드**: Cloudflare Workers/Vercel Functions + Postgres(Neon/Supabase) 또는 D1.
- **키 발급 서비스**: 결제 웹훅 → 서명(개인키는 서버 KMS/시크릿) → 키 반환/이메일. 클라이언트는 기존 오프라인 검증 재사용(추가 개발 최소).
- **결제**: 국내는 **KCP/PortOne**(저장소에 PayKit MCP 연결됨 → 프로토타입 활용), 글로벌은 Stripe/Paddle(Paddle는 세금·MoR 처리 이점).
- **업데이트 배포**: GitHub Releases(무료) → electron-updater.
- **크래시/분석**: Sentry(이미 배선) + 제품분석은 self-host 가능한 PostHog.

## D-4. 게이트웨이(프록시) 도입 시 고려
- 장점: 사용량 과금 수익, 키 관리 사용자 부담↓, 모델 추상화.
- 단점: LLM 원가 선지출·남용·레이트리밋·정산·가용성 책임. **크레딧 시스템**·쿼터·비용 상한 필수.
- 권장: **BYOK를 기본**으로 유지하고 게이트웨이는 상위 플랜 옵션으로.

## D-5. 서버측 데이터 모델(도입 시 최소)
```
users(id, email, created_at)
licenses(id, user_id, edition, employees[], expires_at, signature, revoked)
purchases(id, user_id, sku, amount, currency, pg_ref, status, created_at)
devices(id, user_id, machine_hash, last_seen)      // 좌석/활성화 제한 시
telemetry_events(id, anon_id, type, props_json, ts) // opt-in, 개인정보 배제
```
- 라이선스는 **서명 검증이 원본 진실**(서버는 발급·폐기 목록만). 오프라인 검증 유지가 핵심 자산.

## D-6. 보안 / 컴플라이언스
- API 키: renderer 노출 금지·safeStorage(이미 준수). 서버는 사용자 LLM 키를 **보관하지 않음**(BYOK).
- 개인정보: 텔레메트리 기본 off·명시 동의·redaction. 국내 **개인정보보호법**, 해외 **GDPR** 대응(수집 최소·삭제 요청 경로).
- 라이선스 개인키: 서버 KMS/HSM 또는 시크릿 매니저. `config/license-private.pem`은 개발용, 프로덕션은 서버 전용.
- 결제: PCI는 PG에 위임(카드정보 비저장).

## D-7. 백엔드 단계적 도입
- **Phase 0 (현재):** 완전 오프라인. 수동 키 발급. → 초기 판매 가능.
- **Phase 1:** 키 발급 API + 결제 웹훅(자동화). 서버리스 1개 함수 + DB.
- **Phase 2:** 자동 업데이트 + opt-in 텔레메트리/크래시 집계.
- **Phase 3:** 팀/좌석·게이트웨이·클라우드 동기화(필요 검증 후).
